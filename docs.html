API Documentation

Production Ready v1.0.3

Welcome to the official developer documentation for SWASTHYA AI CORE. This high-performance gateway provides an asynchronous orchestration system designed to abstract complex healthcare intelligence and discovery actions into clean, reliable, and predictable web structures.

API base URL : https://swasthya-ai-core.onrender.com

SWASTHYA AI CORE handles multi-pass conversational parsing, state segregation, multi-source hospital discovery via Google Maps Places API and Tavily Web Search, asynchronous pipeline execution via AsyncIO Task Executor, and task state management via a Redis backend. This abstracts all complex distributed workflows into simple web primitives for smooth client integrations.

Read Workflow View Repository

Architecture

SWASTHYA AI CORE uses a fully asynchronous, state-segregated design. FastAPI handles all HTTP requests. Background discovery jobs run in-process via AsyncIO task scheduling (asyncio.create_task) — no external message broker is required. Live task progress is stored in Redis and polled by the client.

Complete Workflow

CRITICAL CONCEPT: The single biggest point of confusion is how the Context API connects to the Discovery API. You must forward the nested context_data object from the Context response directly into the Discovery request. Do not manually restructure it.
1

Context Analysis Loop

Submit raw text blocks into the Context API. If the payload response establishes is_context_sufficient == false, block lookups, capture the generated followup_question, and update your conversational view.

2

Discovery Job Handshake

Once sufficiency conditions pass, extract the compiled, nested context_data payload block and map it directly into the Discovery gateway interface. This returns a task trace key immediately.

3

State Progress Polling

Track real-time background orchestration milestones by querying the transient Task Progress engine until status traces mark operational completion.

Complete End-to-End Javascript Example

const API_BASE = "https://swasthya-ai-core.onrender.com";

async function findHospitals(patientMessage) {
    // 1. Analyze Context with Core Models
    const contextRes = await fetch(`${API_BASE}/context/analyze`, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ message: patientMessage })
    });
    const contextResponse = await contextRes.json();

    // Verify context state conditions before scheduling lookup
    if (!contextResponse.is_context_sufficient) {
        console.log("Ask User: " + contextResponse.followup_question);
        return { status: "incomplete", prompt: contextResponse.followup_question };
    }

    // 2. Queue Asynchronous Discovery using exact context_data payload block
    const discoveryRes = await fetch(`${API_BASE}/discovery/search`, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ 
            context: contextResponse.context_data, // <-- CRITICAL STEP
            max_results: 4 
        })
    });
    const { task_id } = await discoveryRes.json();

    // 3. Poll Transient Progress Engine increments
    return new Promise((resolve, reject) => {
        const interval = setInterval(async () => {
            const progressRes = await fetch(`${API_BASE}/tasks/${task_id}/progress`);
            const progress = await progressRes.json();
            
            console.log(`Progress: ${progress.progress_percent}% - ${progress.current_stage}`);
            
            if (progress.status === "completed") {
                clearInterval(interval);
                resolve(progress.result.recommendations);
            } else if (progress.status === "failed") {
                clearInterval(interval);
                reject(progress.error_message);
            }
        }, 2000); // Poll systematically every 2 seconds
    });
}

Context API

Processes raw conversational string sequences to identify patient symptoms, clinical severity markers, and implicit or explicit geographical boundaries.

POST https://swasthya-ai-core.onrender.com/context/analyze

Request Body (application/json)

FieldTypeRequiredDescription
message string ✅ Required The raw, unparsed message string supplied by the patient.
context_id string | null Optional Multi-turn conversations. Pass the context_id returned by a previous response to continue the same conversation. Omit for a fresh conversation (fully backward compatible).
correlation_id string | null Optional An optional tracing identifier useful for correlating backend distributed operations.

Response Lifecycle Payloads

SWASTHYA AI CORE enforces dynamic payload compression. If incoming messages are simple greetings (e.g., "Hello") or require detail aggregation ("I'm sick"), heavy nested structural models (like context_data) are omitted entirely to save runtime throughput.

Response Fields

FieldTypeRequiredDescription
context_id string ✅ Required The unique cryptographic execution identifier key generated by backend routing managers. Use this to continue conversations.
session_version integer ✅ Required Increments whenever clinical context changes. Helps frontend detect updated recommendations. Used for conversation versioning.
is_greeting boolean ✅ Required Flags whether incoming user content mirrors non-actionable opening phrases.
is_context_sufficient boolean ✅ Required CRITICAL GATEKEEPER. If evaluated as false, block processing and trigger the followup string. If true, initiate discovery pipelines immediately.
needs_followup boolean ✅ Required Evaluates whether missing data sets require a chat-driven patient request.
followup_question string | null Optional Deterministic, contextually targeted string to display directly to patients. Omitted or null when context needs match sufficiency benchmarks.
missing_fields array[string] ✅ Required Tracks missing target fields like ["location", "symptoms"] to support custom diagnostic views.
context_data object | null Optional CRITICAL INJECTION PAYLOAD. Returned exclusively when sufficiency triggers evaluate as true. Contains multi-layered nested data nodes mapping specific analytical flags.

Field Usage & Client Action

What do I do with these fields?

  • context_id: Returned by Context API → Send back to Context API for follow-up messages → Also embedded inside context_data → Finally forwarded into Discovery API.
  • session_version: Returned by Context API → Frontend stores it → If it changes, recommendations may have changed.
  • context_data: Returned by Context API → Do NOT modify → Pass directly into Discovery API under "context": context_data.
Client Action Box
If is_context_sufficient == false:
↳ Display followup_question → Collect user's answer → Call Context API again → Pass previous context_id.

If is_context_sufficient == true:
↳ Extract context_data → Call Discovery API.

Example Request

{
  "message": "I have severe chest pain and I live in South Delhi",
  "correlation_id": "optional-trace-id-123"
}

Discovery API

Accepts the nested object structure extracted via parsing runs to instantly append background lookup items inside active cluster task worker queues.

POST https://swasthya-ai-core.onrender.com/discovery/search

Request Body (application/json)

FieldTypeRequiredDescription
context object ✅ Required The raw, unedited nested context_data block extracted out of previous Context API verification lookups.
max_results integer Optional Sets constraints on total returned target items. Defaults to 4 items. Constraints bound execution ranges between 2 and 10 results.
correlation_id string | null Optional An optional tracing identifier useful for correlating backend distributed operations.

Response Fields

FieldTypeRequiredDescription
task_id string ✅ Required The unique cryptographic execution identifier key generated by background task routing managers. Use this to poll live worker status blocks.
status string ✅ Required Initial state condition placeholder flag. Evaluates systematically as "queued" upon instant endpoint resolution.
message string ✅ Required Informational message confirming the background task was queued.

Field Usage & Client Action

What do I do with these fields?

  • task_id: Returned by Discovery API → Poll GET https://swasthya-ai-core.onrender.com/tasks/{task_id}/progress until status becomes completed.
Client Action Box
After Discovery API:
↳ Store task_id → Begin polling Task endpoint every 2 seconds until status == "completed".

Task Progress API

Reads transient progress parameters tracked inside in-memory data caches, enabling live operational visibility over background collection processes.

GET https://swasthya-ai-core.onrender.com/tasks/{task_id}/progress

Response Fields

FieldTypeRequiredDescription
status string ✅ Required State enumeration strings tracking background worker lifecycles: "queued", "processing", "completed", or "failed".
progress_percent integer ✅ Required Numerical state counter tracking compilation progress between 0 and 100 percent. Maps seamlessly to client UI progress indicators.
current_stage string ✅ Required Descriptive tracing text blocks detailing active steps (e.g., "Researching...", "Scraping provider data portals via Playwright...").
result object | null Optional Defaults to null during active worker execution. Appends rich dataset models once status properties flag explicit operational completion.
Client Action Box
After Task Endpoint:
↳ If status == "completed" → Stop polling → Display recommendations from result.recommendations.

Detailed Recommendation Model Schema

Each object inside result.recommendations is a RankedHospital with the following fields:

FieldTypeDescription
rank integer 1-based position in the recommendation list. Rank 1 is the best match for the patient's clinical context.
hospital_name string Canonical, deduplicated hospital name. Department prefixes, SEO noise, and city suffixes are stripped during entity resolution.
hospital_type string Facility classification: government, private, trust, or charitable.
formatted_address string | null Full street address from Google Maps Places API (e.g., "B-22, Sector 62, Noida, Uttar Pradesh 201309"). Null if Google Maps did not match the hospital.
latitude float | null WGS-84 latitude from Google Maps Places API. Use with longitude to render map markers or compute "Get Directions" links.
longitude float | null WGS-84 longitude from Google Maps Places API.
google_maps_place_id string | null Google Maps stable Place ID (e.g., ChIJ...). Use to link directly to the hospital's Google Maps page or embed a static map tile.
google_maps_url string | null Direct URL to the hospital's Google Maps listing. Can be used as-is for a "Get Directions" button.
distance_km float | null Haversine straight-line distance in kilometres from the patient's location. Null if patient coordinates or hospital coordinates are unavailable.
estimated_travel_time_minutes integer | null Rough travel time estimate assuming 25 km/h average city speed (~2.4 min/km). Not real-time traffic. Null if distance is unavailable.
contact_number string | null National phone number from Google Maps (e.g., "+91-1244962200"). Null if Google Maps did not return a phone number for this place.
website string | null Official hospital website URL. Sourced from Google Maps Places API or the Tavily search result URL.
accreditations array[string] Quality certifications detected. Values include NABH, JCI, NABL. Empty array if none detected.
overall_rating float | null Google Maps aggregate rating (1.0–5.0). Null if no Maps match or the place has fewer than 5 reviews.
review_count integer | null Total number of Google Maps user reviews. Higher values indicate more reliable rating data.
has_emergency boolean True if 24/7 emergency or casualty services were detected from scraped website content. False if not detected (does not mean absent).
has_icu boolean True if ICU, NICU, PICU, or CCU was detected in scraped website content.
estimated_cost_range object | null Nested cost object: { "min_inr": 5000, "max_inr": 50000, "currency": "INR" }. Estimated from LLM analysis of scraped content. Treat as a rough guide, not a binding quote. Null if no cost data was found.
overall_score float Composite ranking score (0.0–1.0). Weighted across clinical suitability (35%), quality (30%), affordability (20%), and accessibility (15%). Emergency mode weights accessibility at 50%.
summary string 2–3 sentence patient-friendly explanation of why this hospital was recommended for the patient's specific condition and location.
pros array[string] Up to 3 key strengths relevant to the patient's query (e.g., "NABH accredited cardiac centre", "24/7 emergency department").
cons array[string] Up to 1 known limitation or caution (e.g., "Long wait times reported", "Potentially high cost for uninsured patients").

Integration Examples — Complete Conversation Lifecycle

Architecture Note (v1.0.3): The Context API maintains server-side conversation memory using Redis (30-minute rolling TTL). The client stores the context_id returned by each response and echoes it back in the next request. The backend preserves all previously extracted clinical data and keeps the session alive after context completion for post-discovery follow-ups. You never need to resend previous messages.
1

Scenario 1 — Greeting

User opens the app and sends a generic greeting. The backend detects it via the deterministic fast path (no LLM call) and returns a localized response in under 1 second.

// Request — no context_id needed for the first message
POST https://swasthya-ai-core.onrender.com/context/analyze
{
  "message": "Hello"
}

// Response
{
  "context_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",  // <-- SAVE THIS
  "is_greeting": true,
  "is_context_sufficient": false,
  "needs_followup": true,
  "followup_question": "Hello! I'm MedPath AI. Please describe your symptoms...",
  "processing_latency_ms": 820
}

Client Action

  • Store context_id in your component state or URL query param. You will need it for every subsequent request in this conversation.
  • Render followup_question in the chat UI as the AI's reply.
  • Wait for the user to type their next message.
2

Scenario 2 — Incomplete Context (Follow-up Turn)

User provides their symptom. The backend loads the stored PatientContext from Redis using the context_id, runs Pass 2 in merge mode, and determines that a location is still missing.

// Request — echo back the context_id from Turn 1
POST https://swasthya-ai-core.onrender.com/context/analyze
{
  "context_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "message": "I have severe abdominal pain"
}

// Response — symptoms are now stored in Redis. Location still missing.
{
  "context_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",  // same ID
  "is_context_sufficient": false,
  "needs_followup": true,
  "followup_question": "Where are you currently located? (City or area)",
  "missing_fields": ["location"],
  "processing_latency_ms": 1240
}

Client Action

  • Keep the same context_id. Do not generate a new one.
  • Render followup_question in the chat.
  • Do not resend "I have severe abdominal pain" again. The backend has it stored in Redis. Only send the user's new reply.
3

Scenario 3 — Context Completed

User provides their location. The backend merges it with the stored symptom data, sufficiency passes, and the response includes the full context_data. The Redis session is kept alive (marked context_completed=true) so that post-discovery follow-up questions (pricing, insurance, appointments) can still access this conversational context for up to 30 minutes.

// Request — only the new piece of information
POST https://swasthya-ai-core.onrender.com/context/analyze
{
  "context_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "message": "I'm near Pari Chowk, Greater Noida"
}

// Response — is_context_sufficient = true. context_data is now populated.
// Redis session is kept alive (context_completed=true). TTL is refreshed.
{
  "context_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "session_version": 1,
  "is_context_sufficient": true,
  "needs_followup": false,
  "context_data": {                // <-- EXTRACT THIS ENTIRE OBJECT
    "context_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "session_version": 1,
    "clinical": {
      "symptoms": ["severe abdominal pain"],
      "patient_location": { "city": "Greater Noida", "raw_location": "Pari Chowk, Greater Noida" },
      "urgency_level": "urgent",
      "is_emergency": false
    }
  },
  "processing_latency_ms": 1080
}

Client Action

  • Extract response.context_data — this is a complete, structured object.
  • Pass it verbatim as the context field of the Discovery API request. No transformation needed.
  • Keep the context_id stored. The session remains alive on the backend for 30 minutes for post-discovery follow-ups.
  • To start a completely new search, simply omit context_id in a future request.
4

Scenario 4 — Discovery Job Submission

Submit the context_data object directly as the context field. Do not wrap the entire Context API response. Do not reconstruct or modify the object. The Discovery API registers a background AsyncIO task and returns a task_id immediately — the actual multi-source hospital search runs in the background.

// Request — forward context_data verbatim from the Context API response
POST https://swasthya-ai-core.onrender.com/discovery/search
{
  "context": <context_data object from Context API>,
  "max_results": 4
}

// Response — immediate. The actual search runs in background.
{
  "task_id": "9f3e7b12-0c4d-4a81-a9b2-f823c6d15e90",
  "status": "queued",
  "message": "Discovery task queued successfully"
}

Client Action

  • Save task_id in component state (or URL, for refresh resilience).
  • Show a loading indicator to the user ("Searching for hospitals…").
  • Immediately begin polling GET https://swasthya-ai-core.onrender.com/tasks/{task_id}/progress every 2–3 seconds.
5

Scenario 5 — Progress Polling

The worker streams real-time updates to Redis at each pipeline stage. Poll every 2–3 seconds. Map progress_percent to a progress bar and current_stage to a status label.

// Poll 1 (0%) — immediately after submission
GET https://swasthya-ai-core.onrender.com/tasks/9f3e7b12.../progress
{ "status": "queued",      "progress_percent": 0,  "current_stage": "Queued"      }

// Poll 2 (10%)
{ "status": "processing",  "progress_percent": 10, "current_stage": "Planning"    }

// Poll 3 (25%)
{ "status": "processing",  "progress_percent": 25, "current_stage": "Searching"   }

// Poll 4 (55%)
{ "status": "processing",  "progress_percent": 55, "current_stage": "Research"    }

// Poll 5 (75%)
{ "status": "processing",  "progress_percent": 75, "current_stage": "Ranking"     }

// Poll 6 (90%)
{ "status": "processing",  "progress_percent": 90, "current_stage": "Explanation" }

// Poll 7 (100%) — FINAL
{ "status": "completed",   "progress_percent": 100, "current_stage": "Completed",
  "result": { "recommendations": [...] } }

Client Action

  • status == "processing" → update progress bar, wait for next poll.
  • status == "completed" → stop polling immediately, render recommendations.
  • status == "failed" → stop polling, display error_message to user.
  • Poll interval: 2–3 seconds. Never poll faster than 1 second.
6

Scenario 6 — Session Expiry (Error Handling)

If the user takes longer than 15 minutes to reply to a follow-up question, the Redis session expires. Sending the stale context_id returns HTTP 410 Gone.

// Stale context_id after 15 minutes of inactivity
POST https://swasthya-ai-core.onrender.com/context/analyze
{
  "context_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "message": "I'm near Pari Chowk"
}

// HTTP 410 Gone
{
  "code": "CONTEXT_EXPIRED",
  "message": "Conversation session 'a1b2c3d4...' has expired. Please start a new conversation.",
  "detail": "Your conversation session has expired. Please start a new conversation by omitting context_id."
}

Client Action

  • Detect HTTP 410 or code == "CONTEXT_EXPIRED".
  • Clear your stored context_id.
  • Inform the user: "Your session expired — please describe your symptoms again."
  • On the next request, omit context_id to start a fresh conversation.
7

Scenario 7 — Post Discovery Context Update

After a successful discovery, a user might provide new clinical information (e.g., "Actually I also have diabetes"). Because the Redis session remains active for 30 minutes, you can send this new message to the Context API with the same context_id. The backend will perform Delta Extraction and increment the session_version.

// Request — sending new information after Discovery has completed
POST https://swasthya-ai-core.onrender.com/context/analyze
{
  "context_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "message": "Actually I also have diabetes"
}

// Response — context_data is updated, and session_version increments
{
  "context_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "session_version": 2, // <-- INCREMENTED
  "is_context_sufficient": true,
  "context_data": { ... }
}

Client Action

  • Compare the returned session_version with your stored version.
  • If session_version increased, it means the clinical context changed.
  • Prompt the user: "Your health profile was updated. Would you like to search for hospitals again?" or automatically re-run the Discovery API with the new context_data.

Optional Behaviour Notes

  • When to send context_id: Send it on every follow-up message to continue an existing conversation.
  • When NOT to send context_id: Do not send it on the very first message of a new conversation, or when the user explicitly clicks "Start Over".
  • When to send correlation_id: Send it if your frontend uses distributed tracing (e.g., Datadog, Sentry) and you want to correlate backend logs with frontend sessions.
  • When NOT to send correlation_id: If you do not have a tracing system, you can safely omit it.
  • When to use session_version: Use it to detect if the patient added new clinical symptoms after you already showed them hospital recommendations, so you can offer to refresh the results.
  • When to ignore session_version: If your UI does not allow users to chat after recommendations are shown, you can ignore this field.

Complete JavaScript Integration (Multi-Turn)

const API_BASE = "https://swasthya-ai-core.onrender.com";

class MedPathSession {
    constructor() {
        this.contextId = null;  // Persists across turns
        this.taskId = null;
        this.pollInterval = null;
    }

    async sendMessage(userMessage) {
        const body = { message: userMessage };
        // Echo context_id if we have one (conversation continuation)
        if (this.contextId) body.context_id = this.contextId;

        const res = await fetch(`${API_BASE}/context/analyze`, {
            method: "POST",
            headers: { "Content-Type": "application/json" },
            body: JSON.stringify(body),
        });

        // Handle session expiry
        if (res.status === 410) {
            this.contextId = null;
            const err = await res.json();
            return { type: "expired", message: "Session expired. Please start over." };
        }

        const data = await res.json();

        // Always update contextId (it's the same value, but good practice)
        this.contextId = data.context_id;

        if (data.is_context_sufficient) {
            // Context complete — hand off to Discovery
            await this.startDiscovery(data.context_data);
            return { type: "searching" };
        } else {
            // Still gathering info — show follow-up question
            return { type: "followup", question: data.followup_question };
        }
    }

    async startDiscovery(contextData) {
        const res = await fetch(`${API_BASE}/discovery/search`, {
            method: "POST",
            headers: { "Content-Type": "application/json" },
            body: JSON.stringify({ context: contextData, max_results: 4 }),
        });
        const { task_id } = await res.json();
        this.taskId = task_id;
        this.contextId = null;  // Context spent — clear it

        // Begin polling every 2 seconds
        this.pollInterval = setInterval(() => this.pollProgress(), 2000);
    }

    async pollProgress() {
        const res = await fetch(`${API_BASE}/tasks/${this.taskId}/progress`);
        const data = await res.json();

        // Update UI with progress_percent and current_stage
        updateProgressBar(data.progress_percent, data.current_stage);

        if (data.status === "completed") {
            clearInterval(this.pollInterval);
            renderRecommendations(data.result.recommendations);
        } else if (data.status === "failed") {
            clearInterval(this.pollInterval);
            showError(data.error_message);
        }
    }
}

Asynchronous Discovery Pipeline

The Hospital Discovery pipeline is fully asynchronous. When POST /discovery/search is called, the server registers a background asyncio.create_task() and returns a task_id immediately. The actual work runs in the background through the following stages:

Discovery Pipeline Stages

  • Planning (10%) — LLM generates a multi-query search strategy from the clinical context.
  • Searching (25%) — Google Maps Places API, Tavily web search, and NABH scraper run in parallel. Whichever source fails is skipped gracefully.
  • Entity Resolution & Dedup (40%) — Canonical hospital names are extracted. Maps and Tavily candidates for the same hospital are merged; Maps-sourced structured data (coordinates, phone, rating) takes priority.
  • Research (55%) — Top-8 shortlisted hospitals are scraped concurrently (max 5 in parallel). A Playwright Browser Pool handles JavaScript-heavy hospital sites. Scraping is bounded by a strict 6-second per-hospital timeout.
  • LLM Summarization (70%) — Gemini batches all shortlisted candidates and extracts specialties, cost ranges, ICU/Emergency flags, and quality signals from scraped text.
  • Ranking (85%) — Multi-criteria scoring across clinical suitability, accessibility, quality, affordability, and confidence. Emergency requests weight proximity 50%.
  • Explanation (90%) — LLM generates patient-friendly summaries, pros, and cons for each ranked hospital.
  • Completed (100%) — Full recommendation bundle written to Redis. Available via polling endpoint.
Performance Expectation: Total discovery latency is typically 20–45 seconds depending on how many hospital websites respond to scraping. Emergency searches (fewer candidates, proximity-weighted) typically complete faster.

Error Handling

All errors follow a consistent JSON shape: { "code": "ERROR_CODE", "message": "...", "detail": "..." }

Status CodeWhen It OccursClient Recovery
400 Bad Request Request body is structurally invalid (e.g., missing required fields, wrong types). Fix the request body. Ensure context is the raw context_data object, not the full Context API response.
404 Not Found Task ID does not exist in Redis (never created, or already expired). Task results expire after 6 hours. If polling a stale task_id, ask the user to retry the search.
409 Conflict A discovery task is already running for this context. Duplicate submission prevented. Poll the existing task_id rather than submitting again.
410 Gone The context_id sent to the Context API has expired (session TTL exceeded). Clear stored context_id. Inform the user their session expired and ask them to describe symptoms again. Omit context_id on next request.
422 Unprocessable Entity Pydantic validation failed — field type mismatch or constraint violation (e.g., max_results out of range). Check field types against the schema table. max_results must be 2–10.
429 Too Many Requests LLM provider rate limit hit. The gateway implements immediate failover to Mistral when Gemini returns 429. Typically self-resolving. Wait 5 seconds before retrying. If persistent, the LLM circuit breaker is open — retry after 60 seconds.
500 Internal Error Unexpected server error. If it occurs mid-pipeline, the task status is set to failed with an error_message. For synchronous 500s, retry after a brief delay. For pipeline failures, read error_message from the progress endpoint and surface it to the user.
503 Service Unavailable Redis is unreachable or the LLM gateway circuit breaker is open across all providers. The backend degrades gracefully — pipeline continues where possible but task state may not be persisted. Retry after 60 seconds.

Known Limitations

  • Estimated costs are approximate. Cost ranges are inferred by an LLM from scraped website text and are not official price lists. Always verify costs directly with the hospital.
  • Distance requires patient coordinates. distance_km and estimated_travel_time_minutes are only populated when the patient's location resolves to a known geocoordinate. City-level locations may not produce precise distances.
  • Ratings depend on Google Maps coverage. Hospitals without a Google Maps listing will have overall_rating and review_count as null.
  • ICU and Emergency flags are heuristic. These are detected from scraped website text using keyword matching, not from a live hospital API. A False value does not confirm the absence of these services.
  • Availability and wait times are not real-time. The system does not connect to hospital bed management or appointment systems. Recommendations are informational only.
  • Travel time is not traffic-aware. estimated_travel_time_minutes uses a fixed 25 km/h average and does not account for traffic, road type, or time of day.
  • These are recommendations, not referrals. SWASTHYA AI CORE is a decision-support tool. Final clinical decisions should always be made in consultation with a qualified healthcare professional.

Production Best Practices

  • Polling Cadence Rules: Enforce strict client boundaries constraining progress checks to intervals between 2 and 3 seconds. Excessive polling frequencies can quickly saturate active connection loops and exhaust distributed caching bandwidth.
  • Pipeline Isolation Boundaries: Keep parsing operations and resource discovery workflows decoupled. If input structures undergo changes, restart validation loops through the Context API first. Do not cycle discovery trackers until sufficiency validations explicitly pass.
  • State Persistence Tracking: Maintain the current operational task_id within your local application tracking state. This ensures that application reloading routines or view recycling events do not disrupt active background lookup tracking.

Frequently Asked Questions

Why are Context extraction operations decoupled from Search routines?

Separating these steps helps optimize system resource usage. Running deep web scrapers or ranking algorithms before a patient specifies their exact symptoms or location wastes significant computing resources. Decoupling helps ensure operations only scale up once clear search requirements are fully established.

How long do background tracking objects persist within system memory?

Task properties maintain data presence within central Redis state stores for a strict 30-minute window post-completion before being automatically cleared out via systematic eviction loops.
